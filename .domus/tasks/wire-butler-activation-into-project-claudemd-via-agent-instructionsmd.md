## What This Task Is

Butler role never activates because the bootstrap chain is broken. The intended flow is:

```
CLAUDE.md → agent-instructions.md → role-activation-rules.md → Butler role → (routes to Taskmaster, Oracle, etc.)
```

What actually happens: CLAUDE.md mentions domus tasks/ideas workflow but never tells Claude to read `.domus/reference/agent-instructions.md`. And even if it did, `agent-instructions.md` covers task lifecycle and CLI usage but never mentions Butler or the staff roles. So the role files sit inert in `.domus/reference/staff/` — correctly built, never loaded.

### What went wrong (observed 2026-03-21, everycent project)

1. User said "let's get going on the budget screen migration"
2. Claude read the task, saw it was `raw` with placeholder acceptance criteria
3. Claude did good research (Angular code, Vue code, Rails API) and had a brief scope discussion
4. Claude then jumped straight into coding instead of recognizing that `raw` status → Taskmaster refinement session
5. User had to intervene twice: first to point out the spec should have been captured in the task body, then to ask why Butler/Taskmaster weren't active
6. Root cause: nothing in Claude's auto-loaded context told it to read the role activation rules or adopt Butler

### The fix

Two changes needed:

**1. Project CLAUDE.md** (generated by `domus init`/`domus update`) needs a line like:
```
Read `.domus/reference/agent-instructions.md` for domus workflow instructions.
```

**2. `agent-instructions.md`** (the template in `src/templates/reference/`) needs a Butler bootstrap section:
```
## Session start
At the beginning of every session, read `.domus/reference/staff/role-activation-rules.md`
and adopt the Butler role. Butler routes intent to the appropriate specialist role
(Taskmaster for refinement, Oracle for ideas, Foreman for dispatch, etc.).
```

This closes the chain: CLAUDE.md (auto-loaded) → agent-instructions.md (read on demand) → role-activation-rules.md → Butler → specialist roles.
