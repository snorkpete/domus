## What This Task Is

Tasks come in two kinds:

- **Feature work** — new features, enhancements, refactors, and bug fixes. The default workstream.
- **Health-check work** — tasks generated by the health-check module, worked on in dedicated sessions.

The health-check module will generate a large volume of tasks that, mixed into the normal `domus task list`, drown out feature work. Use the existing `tags` field to mark health-check tasks, teach `task list` / `overview` to filter by tag, and hide the `health-check` tag from the default view — mirroring how `wont-fix` is already handled.

Note: bug fixes are **feature work**, not health-check work. The split is by origin (where the task came from), not by what kind of change it is.

Scope note: no new schema field. Tasks already have `tags: string[]`. This task wires tag-based filtering into the list/overview commands and establishes `health-check` as a default-hidden tag.

## Design Decisions (resolved during refinement)

- **Default-hidden mechanism:** config-driven. Add `defaultHiddenTags: string[]` to `.domus/config.json` (seeded with `["health-check"]` by `domus init`). Read at filter time by `list` and `overview`.
- **Flag surface:** `--tag <t>` and `--exclude-tag <t>`. Both cumulative (pass multiple times, or comma-separated — match existing `--tags` parsing convention on `task add`). Config's `defaultHiddenTags` auto-adds to the exclude set in the background.
- **Include > exclude precedence:** `--tag` is a whitelist; `--exclude-tag` is a blacklist. If a task matches **any** `--tag`, it is shown — exclude rules (CLI or config) are ignored for that task. Examples:
  - `--tag a --exclude-tag b`, task has both `a` and `b` → shown (include wins).
  - `--tag a --exclude-tag a`, task has `a` → shown (include wins, even when same tag).
  - No `--tag` provided → exclude set (CLI + config) is the only filter.
  - Practical consequence: `--tag health-check` surfaces health-check tasks despite the config default-hide, with no special-casing needed.
- **Overview vs list:** share filter logic. `list` is Claude's flat view; `overview` is the human's grouped view. Both apply the same tag filtering before grouping/listing. Extract the filter into a shared helper.

## Acceptance Criteria

- [ ] `.domus/config.json` supports `defaultHiddenTags: string[]`. `domus init` seeds it with `["health-check"]`. `config.ts` loader reads it; missing field defaults to `[]`.
- [ ] `domus update` migrates existing projects: if `defaultHiddenTags` is missing from an existing `config.json`, add it with `["health-check"]`. If present, leave it alone (don't clobber user edits).
- [ ] `task list` and `task overview` both support `--tag <t>` and `--exclude-tag <t>`, cumulative (repeatable or comma-separated).
- [ ] Filter semantics:
  - If `--tag` set is non-empty: task shows iff it has at least one tag in the `--tag` set. Exclude rules are ignored for include-matches.
  - If `--tag` set is empty: task is hidden iff it has at least one tag in the effective exclude set (CLI `--exclude-tag` ∪ config `defaultHiddenTags`). Otherwise shown.
- [ ] Filter logic is extracted into a shared helper (e.g. `src/commands/task/helpers.ts` or `src/lib/task-filters.ts`) used by both `list` and `overview`.
- [ ] `health-check` tag is hidden from the default `task list` / `overview` output. Running `task list --tag health-check` surfaces them.
- [ ] Tests cover: default hiding via config, `--tag` overrides config hiding, `--tag` overrides CLI `--exclude-tag` (include > exclude), cumulative flags, and a task with no tags (shown when `--tag` is empty, hidden when `--tag` is non-empty).

## Implementation Notes

- Config lives in `src/lib/config.ts` (loader) and `src/commands/init.ts` (seeder). Add the new field in both.
- Migration for existing projects: add a step in `src/lib/update-steps.ts` (called by `src/commands/update.ts`) that reads `.domus/config.json` and adds `defaultHiddenTags: ["health-check"]` if the field is missing. Idempotent — don't overwrite if present.
- Filter logic lives in `src/commands/task/list.ts` and `src/commands/task/overview.ts` today. Extract the shared predicate — both commands call it after loading tasks, before formatting.
- Flag parsing: use `parseFlag` / `hasFlag` from `src/lib/args.ts`. For cumulative flags, either allow repeated `--tag x --tag y` or `--tag x,y` — match the `--tags` parsing already in `task add` (comma-split on a single flag value) for consistency.
- `task add` and `task update` already accept `--tags a,b` — no changes needed there.
- Backfill is not scripted: no health-check tasks exist today. If any surface before the health-check module lands, use `task update <id> --tags health-check` manually.
- Commit scope: single commit is the goal. Touches config loader, init seeder, update migration step, two command files, one shared helper, and tests.
