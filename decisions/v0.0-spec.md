# Domus v0.0 Specification

**Date:** 2026-03-20
**Status:** decided

This is the canonical reference for what domus v0.0 is and how it works. When in doubt about any design decision, this spec wins.

---

## Core Principles

1. **Decision/execution split** — the human owns the *what* (desired future state) and the *how* (path to get there). Domus handles execution once the how is nailed down.

2. **The pipeline** — idea → refined tasks → autonomous execution → done. Each stage has a role. Domus assists the human through the decision stages (Oracle, Taskmaster) and takes over for execution (Worker, Foreman).

3. **Domus assists, human drives** — in the decision stages, domus is the co-driver (rally navigator). It provides information, asks clarifying questions, helps refine — but the human makes the calls. As domus gains capability (specs, gap analysis), it can assist more, but the human remains in the driver's seat.

4. **Context mobility** — all state in `.domus/`, committed to git, so any session or worker can pick up any task without briefing. The substrate that makes everything else possible.

---

## What Domus Is

Domus is three things:

1. **The store** — a `.domus/` directory committed alongside code. Contains: tasks, ideas, execution logs, audit log, config, role files, agent instructions.

2. **The CLI** — the `domus` command that manages the store. Commands: `init`, `task {add, status, update, show, start, log, advance, cancel, defer, reopen, overview, ready, list, watch}`, `idea {add, status, update, show, overview, list}`, `dispatch`.

3. **The roles, skills, and context files** — guide Claude on how to work with the system. Role files, skills, CLAUDE.md integration, agent instructions.

---

## Domus is Project-Scoped

Domus operates per-project. There is no workspace model, no project registry, no multi-project coordination. Each project has its own `.domus/` directory. Period.

## Task Lifecycle

A single status field tracks where a task is:

```
raw → proposed → ready → in-progress → ready-for-human-review → done
```

- **raw** — captured, needs refinement work with Taskmaster
- **proposed** — Taskmaster thinks it's worker-ready (execution steps + acceptance criteria written), needs human review
- **ready** — human approved, waiting for dispatch
- **in-progress** — being worked on
- **ready-for-human-review** — Worker committed; awaiting human review and merge by Housekeeper
- **done** — merged into `ready-for-master` and closed

Escape hatches from any state: `cancelled`, `deferred`. Re-entry: `reopen` returns to `raw`.

`ready-for-senior-review` exists in code as a reserved status for a future AI-review step. It is reachable via the Doctor power tool but not part of the advance flow yet.

### The `autonomous` flag

Boolean on each task, always present in frontmatter (`**Autonomous:** true` or `**Autonomous:** false`). Default `true` when transitioning to `ready` in v0.0. Controls whether Foreman/Herald can auto-dispatch. Future: human sets to `false` for tasks they want to execute themselves. Always explicit — never absent.

### No refinement field

The old refinement lifecycle (raw/proposed/refined/autonomous) is fully replaced by the status field above. There is no `refinement` field on tasks.

## CLI Command Model

- **`advance <id>`** — the primary forward transition. Caller doesn't specify target state; the state engine decides. Engine: raw → proposed → ready → in-progress → ready-for-human-review → done.
- **`cancel <id>`, `defer <id>`, `reopen <id>`** — named escape hatch commands with optional `--note`.
- **`status <id> <value>`** — raw power tool for Doctor/housekeeping only. Not used in normal workflow.
- **`dispatch <id>`** — validates task is `ready` + `autonomous: true`, then the dispatch skill handles the rest.

All forward transitions go through `advance`. Direct `status` calls are for exceptional/corrective use only.

### State engine architecture

There is one implementation of task status update logic. All commands — `advance`, `cancel`, `defer`, `reopen`, `status` — go through it. The internal state engine determines how tasks transition (what the next status is) based on: which method was called, the current status of the task, and (in the future) other factors. This state engine logic is self-contained and is the sole source of truth for determining the next status. No command bypasses it.

## The Store (`.domus/`)

```
.domus/
  config.json              ← { root, branch } — project config
  audit.jsonl              ← global audit log, git-ignored
  tasks/
    tasks.jsonl            ← task metadata index
    *.md                   ← individual task files
  ideas/
    ideas.jsonl            ← idea metadata index
    *.md                   ← individual idea files
  execution-logs/
    *.md                   ← one per task, append-only, version-controlled
  reference/
    agent-instructions.md  ← workflow instructions (@-included in agents.md)
    staff/
      role-activation-rules.md  ← thin routing table, always in context
      roles/
        *.md               ← individual role files, loaded on demand
```

### Double log (`domus task log`)

`domus task log <id> <message>` writes the same entry to two destinations:

1. **Execution log** (`.domus/execution-logs/<id>.md`) — version-controlled, per-task, used for progress mobility (next worker reads it to resume)
2. **Audit log** (`.domus/audit.jsonl`) — git-ignored, global, used for operational visibility

One command, two consumers. Workers call `domus task log` for everything — progress updates, decisions, blockers. Blocker notes go into the execution log (not a separate file).

## Roles

Roles are responsibility boundaries, not single files. A role can be composed of: a role file (loaded on demand), skills (trigger descriptions always in context, implementations on demand), and agent instructions (always in context).

### Interactive roles (personas — take over the session)

- **Oracle** — explores ideas through questions. Ask, don't prescribe.
- **Taskmaster** — refines tasks through two-phase (what/how) conversation. Transitions: raw → proposed (after refinement pass), proposed → ready (after human approval).
- **Doctor** — checks store health interactively. Owns `domus task status` as power tool for fixing bad state.

### Always-present roles (thin activation triggers in context, full role on demand)

- **Butler** — auto-injected at session start. Routes to interactive personas based on human intent. Handles role switching mid-session (experimental — session restart is the safe fallback).
- **Herald** — surfaces signals at natural boundaries (session start, "what's next"). Checks for stalled tasks, undispatched ready+autonomous tasks, cold ideas, queue status. Context size awareness is aspirational (depends on Claude Code capability).
- **Foreman** — execution pipeline management. Dispatch (via subagent + worktree), advance, merge and close. In v0.1 becomes active orchestrator routing between worker types.

### Non-interactive

- **Worker** — autonomous task executor. Runs as subagent with `isolation: "worktree"`. In v0.0: single worker handles implement + review + merge. In v0.1+: specializes into worker types (coder, reviewer) dispatched by Foreman.

## Autonomous Execution (v0.0 dispatch path)

1. Human says "dispatch this task" → dispatch skill triggers
2. Validates: task status is `ready`, `autonomous` flag is `true`
3. Skill launches Worker subagent with `isolation: "worktree"`, passes task ID and `--root` path
4. Worker (in worktree) calls `domus task start <id> --branch <branch>` (using `--root`)
5. Worker reads task file + execution log, works through acceptance criteria
6. Worker logs progress via `domus task log` (using `--root`) — each call writes to both execution log and audit log
7. Worker calls `domus task advance` when done → moves to `done` in v0.0
8. Subagent ends, control returns to main session

Workers use `--root` flag to write to the main repo's `.domus/`. No environment variables.

---

## Interactive Sessions

Human opens Claude Code. Butler logic is auto-injected via CLAUDE.md/skills/agent-instructions. Herald and Foreman have thin activation triggers always in context. Butler routes to Oracle/Taskmaster/Doctor based on intent, loading the full role file on demand.

No CLI commands launch interactive sessions. No `domus work`. No `domus idea refine`.

---

## Progressive Disclosure

```
role-activation-rules.md (routing — always in context)
  → role file (operational — loaded when role activates)
    → ADR (background — loaded on demand)
```

Once something is loaded into a Claude session, it stays permanently. The always-loaded budget is precious — keep the routing table thin.

## What v0.1 Adds

1. **`ready-for-senior-review` status activates** — AI senior reviewer slots in *before* `ready-for-human-review`. Advance routes: in-progress → ready-for-senior-review → ready-for-human-review → done.
2. **Worker specialization** — coder and reviewer as distinct worker types with different prompts
3. **Foreman as active orchestrator** — coder finishes → advance → Foreman in main session launches senior reviewer
4. **`send-back` and `merge-and-close`** — CLI commands plus skills that wrap them in natural language. Human uses these via Foreman.
5. **Herald auto-dispatch** — checks for ready+autonomous tasks at natural boundaries and triggers dispatch

> Note: `ready-for-human-review` shipped ahead of v0.1 schedule (2026-04-14) — it is the human gate between Worker commit and Housekeeper merge.
